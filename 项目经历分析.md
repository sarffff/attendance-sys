## 一、项目概述

**项目名称：** LeaveFlow 智能请假与审批平台（企业考勤管理系统）

**项目简介：** 面向企业场景的全流程考勤与请假审批管理平台，覆盖请假申请、多级审批、电子签名、月度台账管理等核心业务流程，支持 8 种角色的细粒度权限控制。

**技术栈：** React 19 + Vite 8 + Redux Toolkit + React Router 7 + Ant Design 6 + Axios

## 四、技术亮点详解（面试准备）

### 1. RBAC 动态路由系统

**做了什么：** 路由和菜单不是静态写死的，而是在用户登录后根据其 `roleCode` 动态生成。

**怎么做的：**
- `menuConfig.js` 中每个菜单项定义了 `roles` 字段（null 表示所有角色可见，或指定角色数组）
- `getRoutes()` 函数根据当前用户角色过滤菜单配置，调用 `createBrowserRouter` 动态创建路由
- Dashboard 和请假历史页面根据角色渲染不同组件（superAdmin vs otherRole）
- ProtectedRoute 守卫组件确保未登录用户无法访问受保护页面

**面试话术：** "我在项目中实现了一套基于角色的动态路由系统。菜单配置集中管理，每个菜单项标注所需角色，用户登录后系统根据角色信息动态过滤菜单并生成路由，实现了前后端统一的权限隔离，新增角色或页面只需修改配置即可，无需改动核心逻辑。"

### 2. 多级审批工作流引擎

**做了什么：** 实现了 6 步请假审批流程，每步绑定不同角色，支持批量操作和动态路由。

**怎么做的：**
- `constantsMap.js` 定义了 `leave_step` 映射，将每个审批步骤与可操作角色关联
- 审批状态机：PENDING -> APPROVING -> APPROVED/REJECTED/CANCELLED
- 第 3 步特殊逻辑：支持选择后续审批人，实现审批路径的动态分叉
- 批量审批/驳回接口，一次操作多个申请

**面试话术：** "审批流程采用了配置化的步骤映射方案，每个步骤与角色绑定，通过常量表驱动而非硬编码。特别在第 3 步实现了动态路由选择，审批人可以指定下一环节的审批领导，实现了流程的灵活性。"

### 3. Canvas 电子签名

**做了什么：** 使用 HTML5 Canvas 实现手写签名采集，支持 HiDPI 屏幕适配。

**怎么做的：**
- 使用 `pointerdown/pointermove/pointerup` 事件监听（兼容触控设备）
- 通过 `devicePixelRatio` 处理高分屏缩放，确保签名清晰度
- Canvas 转 PNG Blob 上传到服务端
- 签名 URL 引用到审批流程记录和 PDF 生成中

### 4. 组件封装与复用

**做了什么：** 封装了 BaseForm 和 BaseTable 两个通用组件。

**BaseForm：** 接收 schema 数组（每项定义 type/field/label/rules/group/span 等），动态渲染表单。新业务只需定义 schema 即可，无需写 JSX。

**BaseTable：** 封装 Ant Design Table，内置分页、数据请求（useFetch）、刷新能力。接收 `request` 函数和 `params`，自动管理数据生命周期。

**面试话术：** "我把各业务模块中重复的表单和列表逻辑抽成了配置驱动的通用组件。BaseForm 通过 schema 描述表单结构，BaseTable 通过传入请求函数自动管理分页和加载状态。新增业务模块时，开发者只需关注配置和业务逻辑，大幅提升了开发效率和代码一致性。"

### 5. 自定义 useFetch Hook

**做了什么：** 封装通用数据请求 Hook，统一管理 loading、error、cleanup。

**怎么做的：**
- 接收请求函数和参数，返回 `{ data, loading, error, refresh }`
- 使用 `useRef` 存储 handler 避免闭包陈旧问题
- isMounted 守卫防止组件卸载后的状态更新
- 配合 Axios 拦截器实现全局认证和错误处理

## 六、可能的面试追问及准备

**Q: 为什么选择 Redux Toolkit 而不是 Context API？**
A: 项目需要跨组件共享用户认证状态（token、userInfo），且需要持久化到 localStorage。Redux Toolkit 的 createSlice 和 createAsyncThunk 提供了标准化的状态管理模式，便于后续扩展（如接入 RTK Query）。

**Q: 动态路由方案有什么优缺点？**
A: 优点是灵活性高，前端权限控制完全配置化。缺点是前端权限可被绕过（修改 localStorage），所以后端也需要做同样的权限校验。实际项目中采用了前后端双重校验。

**Q: 电子签名为什么用 Canvas 而不是第三方库？**
A: 签名需求相对简单（只需手写输入和上传），用原生 Canvas API 实现更轻量，无需引入额外依赖。同时 Pointer Events API 天然支持触屏设备，兼容性好。

**Q: 如何处理组件卸载后的异步请求？**
A: useFetch 中使用了 isMounted 标记（useRef），在 cleanup 函数中设置为 false，异步回调中检查该标记后再更新 state，避免内存泄漏和 React 的 "Can't perform a React state update on an unmounted component" 警告。

**Q: 为什么用内联样式而不是 CSS Modules？**
A: 项目初期为了快速迭代选择了内联样式（JS style objects），便于组件级别的样式隔离。Ant Design 提供了大量内置样式，减少了自定义 CSS 的需求。后续可考虑迁移到 CSS Modules 或 Tailwind CSS 以获得更好的维护性。
